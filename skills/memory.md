# Work Summary Skill

在工作任务完成后，生成一份结构清晰、内容简洁的工作总结，便于归档、回顾和交接。

---

## 输出格式

总结报告分为三个固定模块，使用 Markdown 输出：

### 1. 工作内容

分点列出本次工作涉及的具体任务，每点 **1～2 句话**，说明"做了什么、为什么做"。

**写作要求：**
- 按执行顺序或逻辑顺序排列
- 语言简洁，避免技术堆砌
- 每点聚焦一个独立任务或阶段

**示例：**
```
- 分析了现有数据库表结构，发现用户表缺少索引导致查询慢。
- 为 `users` 表新增了 `email` 字段的唯一索引，并更新了相关查询逻辑。
- 编写并执行了回归测试，验证修改未影响现有功能。
```

---

### 2. 修改文件

列出本次工作中新增、修改或删除的文件，并说明每个文件的改动内容。

**格式规范：**
- 文件名使用代码格式（反引号包裹）
- 每个文件对应一条说明，描述"改了什么、改动目的"
- 若文件较多，可按模块或目录分组

**示例：**
```
| 文件名 | 修改内容 |
|--------|----------|
| `db/schema.sql` | 新增 `email` 字段唯一索引 |
| `src/user_query.py` | 优化查询语句以利用新索引 |
| `tests/test_user.py` | 新增索引相关的回归测试用例 |
```

---

### 3. 成果结论

分点总结本次工作的实际成果，每点 **1～2 句话**，尽量附带可量化的数值指标。

**写作要求：**
- 聚焦"交付了什么、达到了什么效果"
- 优先使用数值（时间、数量、百分比、体积等）体现成果
- 若无数值，则描述质量层面的改进

**示例：**
```
- 查询性能提升约 60%，平均响应时间从 420ms 降至 170ms。
- 新增 8 条回归测试，测试覆盖率从 73% 提升至 81%。
- 数据库结构文档已同步更新，便于后续维护。
```

---

## 完整报告模板
```markdown
## 工作总结

### 工作内容
- ...
- ...

### 修改文件
| 文件名 | 修改内容 |
|--------|----------|
| `...` | ... |

### 成果结论
- ...
- ...
```

---

## 注意事项

- **简洁优先**：每个模块不超过 10 条，超出时合并相关内容。
- **客观准确**：成果描述基于实际数据，不夸大、不模糊。
- **无数据时的处理**：若无可量化指标，用"完成"、"修复"、"优化"等动词描述质量成果，避免空泛表述如"效果良好"。
- **文件未变动时**：若本次工作未涉及文件修改（如调研、分析类任务），"修改文件"模块可注明"本次工作无文件改动"。
