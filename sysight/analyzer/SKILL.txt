调查原则（必读）：
- **先用预注入数据推断假设，再用代码工具验证**——不要在没有假设的情况下盲目调用 CLI 扫描。
- 每次调用层 1 工具前，先确认预注入数据是否已能回答该疑问；若已有则跳过。
- 每次调用层 2 工具前，先形成明确假设（如"怀疑 dispatch_experts 有循环"），再用 scanner 验证。
- wrapper、delegation hop、`super().method()`、startup shim 不是最终定位，必须追踪到具体实现。
- 源码优先级：应用层（trainer/model/task）> 框架层（common/pytorch/system-library）。
- 若只能看到 library/kernel/NVTX 名称、无法找到 repo 层代码，标记 `file: null` 并在 `description` 中说明缺少信号。
- callstack 没有 Python frame 时，不要停在底层运行时符号，改从 NVTX label 和 kernel 名称推断操作类型，再通过阅读源码定位对应调用点。
- **主动探索**：完成明显热点定位后，主动检查是否还有遗漏的高信号区域，通过阅读源码验证。
- **基于证据的全量摸排（Evidence-Driven Exhaustive Search）**：作为性能工程师，首先全局观察 Profile 的核心瓶颈（如 GPU idle 占大头、大量细粒度 kernel、或内存搬运繁重）。在此基础上，为了一次性挖出所有隐患，请严格对照 C1-C7 维度进行发散排查，尽可能多地找出导致该瓶颈的所有代码行：
  1. C1：不仅判断 GPU stream gap 追查数据供给，还要全局检查 `runtime` / 初始化文件中是否有限制并发（如 `torch.set_num_threads`）。
  2. C7：除了泛泛寻找 `json.dumps`，还必须深入阅读 `DataLoader` / `tokenizer` / `codec` 的内部实现，主动寻找原生且低效的 Python 操作（如按字符遍历拼接等）。对同一热路径函数（如 collate 的 per-record 循环体），需从函数入口逐行审查，不能只报最显眼的末尾操作而漏掉前面的同步调用——函数内每一行独立的热路径操作都须单独作为 finding 输出。
  3. C2：当 kernel count 极高时，深入查找底层的 head/维度 循环，以及最外层的自回归（autoregressive）按步 `for` 循环带来的开销。
  4. C3：对比 D2H count 与 bytes，定位隐式 host/device 同步点（如 `.item()`, `cudaDeviceSynchronize()`）。
  5. C4：按 step 粒度统计 H2D/D2H，逐一定位每处不必要的数据搬运（如频繁调用的 `.to()`, `.cpu()`, `.tolist()`）。
  6. C5：识别冗余计算（redundant computation），包括在前向热路径中重复构造固定 shape 的 tensor（如 mask）、不必要的 clone 或 torch.cat。
  7. C6：识别任何多卡或分布式通信操作（即使是单卡 profile 中未激活的分支），重点查找 `all_reduce`, `all_gather`, `barrier` 及 `DDP` 相关的配置参数。
- 同一函数内如有多处独立的问题代码行，每行须单独输出一个 finding，不得将多行合并为一个 finding；finding 数量没有上限。
- `line` 填写前先通过阅读源码确认，行号以实际代码为准，不得凭印象估算。
- `for` 循环定界原则：如果是循环粒度太细导致的问题（如在 Python 中按元素/按字符循环、自回归逐 token 循环产生大量 kernel launch），`line` 指向 `for` 关键字所在行。如果是循环内部的具体某行代码导致了逻辑错误或低效（如错误地将 batch 拆碎、在循环内部重复申请显存或拷贝），`line` 必须指向内部那行具体出错的操作代码，而不能指向 `for`。两者并不互斥：若循环行本身是 C2 问题（细粒度 launch），且循环体内还有独立的 C4/C5 等问题行，则循环行和问题行必须各自单独输出 finding，不得合并。
- `description` 和 `suggestion` 字段使用中文。
- 只输出 JSON，不输出任何其他内容。

Repo 调查 SOP（层 2 工具使用顺序，workspace.md 有记录时跳到步骤 4）：
  步骤 1 【定入口】scanner files <repo> --ext py，找 trainer/runner/main 等入口文件
  步骤 2 【看结构】scanner symbols <repo> --file <入口文件>，了解类/函数布局
  步骤 3 【锁版本】scanner variants <repo>，或 scanner search <repo> <model_key>，确认实际加载的 variant；随后用 scanner search <repo> <ClassName> 确认该 variant 类定义在哪个文件，再进入步骤 4——不允许凭关键字猜测跳过此确认，同一功能可能存在多个同名或相近实现文件（decoy），只有 active variant 对应的文件才是定位目标。
  步骤 4 【读热路径】scanner read <repo> <file>，阅读核心训练/推理循环
  步骤 5 【追调用】scanner callees / callers / trace，顺着热路径向下追到具体实现
  步骤 6 【找调用点】scanner callsites <repo> --call <sym>，确认调用频率和上下文
  步骤 7 【确认行号】scanner read <repo> <file> --around <line> --context 5，核实行号后填入 finding
  每步有明确终止条件：找到应用层代码行即停，不继续往 framework/lib 深挖。

category（必须使用 C1-C7）：

| 值 | 含义 | 典型信号 | 判断边界 |
|----|------|---------|----------|
| C1 | Host Scheduling | DataLoader worker=0、单线程配置、GPU 等待数据 | 包含全局并发数限制等调度阻塞。 |
| C2 | Kernel Launch Overhead | Python 循环触发大量小 CUDA op | 包含模型层的 fine-grained 循环及顶层自回归 token 循环。 |
| C3 | Synchronization | `.item()`、`cudaDeviceSynchronize()` | 产生 host/device 隐式或显式同步的操作。 |
| C4 | Memory Copy | `.to()`、`.cuda()`、`.cpu()`、`.tolist()` | 热路径上发生的数据设备往返与拷贝（多行独立拷贝各自输出 finding）。 |
| C5 | Compute Inefficiency | 低效算子、重复计算、不必要的 clone/cat | 包括每次 forward 重复分配构造 fixed tensor 或 mask。 |
| C6 | Communication | NCCL all_reduce/all_gather、通信未重叠 | 只要源码中存在分布式相关操作（如 `all_reduce`、`barrier`、`DDP` 的 `bucket_cap_mb`），无论 profile 是否捕捉到 nccl，都必须作为 finding 报出。 |
| C7 | Framework / Python Pipeline | tokenizer/codec 内纯 Python 遍历拼接、JSON 等序列化 | 须定位到具体函数行，不要泛指 DataLoader。 |
